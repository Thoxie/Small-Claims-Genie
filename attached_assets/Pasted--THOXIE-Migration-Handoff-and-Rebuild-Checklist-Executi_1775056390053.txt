# THOXIE Migration Handoff and Rebuild Checklist

## Executive Summary

This document is a formal migration handoff for the current THOXIE codebase. It is written for a developer or team that will rebuild, migrate, or stabilize the application in a new environment.

The most important conclusion is that the current canonical application is a **server-backed California small-claims workflow app** built in **Next.js**. It supports structured case intake, document upload, text extraction, evidence-grounded AI chat, filing guidance, and a case dashboard. It is not a mature multi-user SaaS platform yet. Access is primarily controlled by a **case ownership cookie**, not a conventional authentication and authorization system.

The repository also contains a substantial amount of **legacy, prototype, and archived material**. Those materials are useful for understanding the product’s history and intended direction, but they should not be treated as the source of truth for a rebuild. The **root Next.js app** is the canonical implementation.

---

## 1. Purpose of This Handoff

This handoff is intended to give another developer enough context to:

* understand what the current application actually does
* distinguish canonical code from legacy or stale code
* preserve the parts of the system that are already working
* identify the main unfinished or fragile areas
* rebuild the system in a cleaner environment without regressing core functionality

This document is both:

1. a **formal migration handoff**, and
2. a **developer-facing rebuild checklist**.

## 3. Product Definition

The current product is best described as:

> A California small-claims preparation and evidence workflow application that helps a user create and manage a case, upload supporting documents, extract and organize evidence, and ask an AI assistant questions grounded in the case record.

It is not currently:

* a broad legal research platform
* a law firm case-management system
* a multi-tenant SaaS with user organizations and staff roles
* a court e-filing system
* a mature collaboration product

## 4. Architecture Overview

### 4.1 Tech Stack

Current canonical stack:

* **Next.js App Router**
* **React 18**
* **Node 20**
* **PostgreSQL** for structured server persistence
* **Vercel Blob** for file storage
* **OpenAI Chat Completions API** for AI responses
* **Mammoth** for `.docx` text extraction
* **pdf-parse** for machine-readable PDFs
* supporting PDF/OCR libraries including `pdf2json`, `pdfjs-dist`, `tesseract.js`, and `@napi-rs/canvas`
* browser-side storage using **IndexedDB** and **localStorage** for some client-local data

### 4.2 High-level system flow

The current application works roughly like this:

1. A user creates or loads a case in the UI
2. Structured case data is saved to PostgreSQL
3. The user uploads evidence documents
4. Documents are stored in Vercel Blob
5. Extracted text and document metadata are stored in PostgreSQL
6. Extracted text is chunked into a chunk table
7. AI chat loads case data plus document/chunk context and answers either deterministically or via OpenAI

### 4.3 Architectural strengths

* server-backed source of truth for case and document data
* evidence-grounded AI design rather than freeform chat only
* deterministic workflow and readiness logic around small-claims preparation
* chunked document storage without needing vector infrastructure
* config-driven jurisdiction and forms logic

### 4.4 Architectural weaknesses

* no real user authentication model
* no true team collaboration model
* scanned-PDF OCR flow is incomplete
* chat history and drafts are local rather than fully server-backed
* duplicate and stale code in the repo increases migration risk
* some technical concerns are handled in a lightweight way that is acceptable for prototype-stage production, but not ideal for scale

## 6. Data Model and Database Design

### 6.1 Source of truth for schema

The canonical database schema is defined in:

* `app/_lib/server/ensureSchema.js`

Do **not** treat `/api/db-init` as the authoritative schema definition. It is stale relative to the current system.

### 6.2 Database tables

#### Table: `thoxie_case`

Purpose:

* stores the canonical case record
* stores lightweight ownership metadata
* acts as the parent record for documents

Key fields:

* `case_id`
* `owner_token_hash`
* `owner_claimed_at`
* `owner_last_seen_at`
* `case_data` (JSONB)
* `created_at`
* `updated_at`

Interpretation:

* the database keeps most business data inside `case_data` rather than fully normalized tables
* ownership is enforced by hashing a browser-held token and comparing it to the stored hash

#### Table: `thoxie_document`

Purpose:

* stores uploaded file metadata
* stores extracted text
* stores OCR lifecycle status
* stores evidence-oriented metadata used in UI and AI retrieval

Key fields:

* `doc_id`
* `case_id`
* `name`
* `mime_type`
* `size_bytes`
* `doc_type`
* `exhibit_description`
* `evidence_category`
* `evidence_supports`
* `blob_url`
* `uploaded_at`
* `extracted_text`
* `extraction_method`
* `ocr_status`
* `ocr_job_id`
* `ocr_provider`
* `ocr_requested_at`
* `ocr_completed_at`
* `ocr_error`

#### Table: `thoxie_document_chunk`

Purpose:

* stores chunked extracted text
* supports evidence-grounded AI retrieval
* stores lightweight structure metadata

Key fields:

* `chunk_id`
* `case_id`
* `doc_id`
* `chunk_index`
* `chunk_text`
* `chunk_kind`
* `chunk_label`
* `section_label`
* `page_start`
* `page_end`
* `char_start`
* `char_end`
* `structural_flags`
* `created_at`

### 6.3 Relationships

* one case has many documents
* one document has many chunks
* deleting a case cascades to documents
* deleting a document cascades to chunks

### 6.4 Structured case JSON (`case_data`)

The case JSON stores most of the application-specific workflow state.

Important categories include:

* case identity and timestamps
* jurisdiction data
* party data
* claim details
* service details
* fee waiver details
* case number and hearing data
* facts and narrative items

This design gives flexibility but places significant validation responsibility in application code rather than in the relational schema.

## 7. API Surface

### 7.1 `/api/case/load` — GET

Purpose:

* load a case by `caseId`
* claim an unowned case on first access
* enforce ownership for already-claimed cases

Returns:

* normalized case payload or ownership error

### 7.2 `/api/case/save` — POST

Purpose:

* create or update structured case data
* enforce ownership via cookie/token system

Inputs:

* `caseId`
* `caseData`

Returns:

* saved normalized case

### 7.3 `/api/documents` — GET / PATCH / DELETE

#### GET

Supports:

* list documents by `caseId`
* load document detail by `docId`
* open blob-backed document content with `open=1`

#### PATCH

Used to update document metadata such as:

* document type
* exhibit description
* evidence category
* evidence supports tags

#### DELETE

Used to delete:

* document DB row
* associated blob object if available

### 7.4 `/api/blob-upload` — POST

Purpose:

* pre-authorize client-side blob upload, mainly for larger PDFs

Used as part of a two-step upload flow, not as the final ingest step.

### 7.5 `/api/ingest` — POST

Purpose:

* canonical document ingestion endpoint
* accepts multipart upload or JSON-backed finalize ingest
* stores document metadata
* extracts text
* chunks text
* dispatches OCR when needed

This is the core upload-processing route.

### 7.6 `/api/ocr/callback` — GET / POST

Purpose:

* receive completion callbacks from an OCR provider or service
* update OCR lifecycle status
* persist OCR text and rebuild chunks when OCR succeeds

### 7.7 `/api/ocr/retry` — GET / POST

Purpose:

* retry OCR for a document that is in a retryable state

### 7.8 `/api/chat` — POST

Purpose:

* core AI assistant endpoint
* loads case, documents, and chunk context
* applies ownership and optional beta restrictions
* chooses deterministic or live-AI response mode

### 7.9 `/api/rag/status` — GET / POST

Purpose:

* provide document/chunk ingestion status for a case

### 7.10 `/api/rag/ingest` — POST

Purpose:

* disabled compatibility route
* currently returns that callers should use `/api/ingest`

### 7.11 Diagnostics and utility endpoints

* `/api/db-ping`
* `/api/health`
* `/api/db-init` (stale)
* `/ai/chat` as compatibility alias to `/api/chat`

## 8. Document Upload and Extraction Pipeline

### 8.1 Current design

The application has two upload paths:

1. **direct multipart upload** for typical files and smaller uploads
2. **blob-backed two-step upload** for larger PDFs

The client orchestration for uploads lives in:

* `app/_repository/documentRepository.js`

### 8.2 Direct upload flow

1. User selects a file in the documents UI
2. Client packages the file with `caseId` and `docType`
3. Client posts multipart data to `/api/ingest`
4. Server extracts text, stores file/blob metadata, stores document row, and writes chunks
5. UI refreshes document status

### 8.3 Blob-backed upload flow for larger PDFs

1. Client creates a `docId`
2. Client requests upload authorization from `/api/blob-upload`
3. Browser uploads file directly to Vercel Blob
4. Client finalizes the upload by sending a JSON-backed ingest request to `/api/ingest`
5. Server reads the blob, extracts text, stores metadata, and writes chunks
6. OCR may be dispatched if needed

### 8.4 Extraction support by file type

#### `.docx`

Uses Mammoth. This is one of the more reliable ingestion paths.

#### machine-readable PDFs

Uses `pdf-parse` as primary extraction path.

#### images

Uses OCR tooling.

#### scanned or image-only PDFs

This is the weakest area. The system detects the problem and is designed to dispatch external OCR, but that external OCR path is not fully finished in the repo.

### 8.5 Storage behavior

For readable documents:

* file stored in blob storage
* metadata stored in `thoxie_document`
* extracted text stored in `thoxie_document.extracted_text`
* chunks stored in `thoxie_document_chunk`

For scanned PDFs awaiting OCR:

* file may still upload successfully
* metadata row may still be created
* extracted text may remain empty
* chunk rows may remain absent
* OCR status fields reflect pending or failed OCR states

### 8.6 Known problems and failure points

This is the main operational risk area.

#### Problem 1: external OCR is architected but not truly complete

The callback/retry/status model exists, but the end-to-end provider implementation is not fully resolved in the repo.

#### Problem 2: inline OCR exists in code but is not the canonical solution

There is evidence of OCR-related work in multiple places, but the main current flow expects an external OCR step for scanned PDFs.

#### Problem 3: successful upload does not guarantee AI-readable evidence

A file can exist in storage yet still provide no extracted text or chunks.

#### Problem 4: the blob upload path is operationally more complex

It is a multi-step flow with more chances for failure than a simple server-side upload.

#### Problem 5: old `.doc` is unsupported

Legacy binary Word docs are rejected.

## 9. AI and Chat Implementation

### 9.1 UI placement

The main chat UI is implemented as a global dock and persistent assistant surface rather than a page-only chatbot.

Important components:

* `src/components/AIChatbox.js`
* `src/components/GlobalChatboxDock.js`

### 9.2 Storage of conversation state

Chat history is currently stored **client-side** in localStorage per case. It is not a full server-side conversation history system.

### 9.3 Server chat flow

The server-side AI flow is implemented in `/api/chat`.

It generally performs the following steps:

1. validate the request and any beta gating
2. load server-authoritative case state
3. load document inventory and relevant chunks
4. classify the request/intent
5. choose a response mode
6. respond via deterministic logic or OpenAI

### 9.4 Model and provider

The codebase is configured around **OpenAI**.

Environment-driven controls determine:

* provider
* model
* timeout

Default model references point to `gpt-4o-mini`.

### 9.5 Context management

The application does not simply send the raw chat thread. It builds structured context from:

* case data
* document metadata
* extracted text
* chunk retrieval results
* evidence summaries
* filing readiness logic

This is one of the stronger design areas in the app.

### 9.6 Retrieval approach

The retrieval strategy is not vector-based. It is more heuristic and lexical:

* chunk text
* metadata
* basic scoring/relevance heuristics

This has tradeoffs:

Pros:

* simpler stack
* easier migration
* more transparent behavior
* no vector DB dependency

Cons:

* weaker semantic recall
* less robust search across paraphrases or loosely-related concepts

### 9.7 Response modes

The chat system supports multiple operational modes, including:

* beta-restricted mode
* rate-limited mode
* readiness or workflow mode
* deterministic evidence response mode
* live AI mode

The system is better understood as a hybrid **workflow/orchestration layer plus LLM** rather than a simple chatbot wrapper.

## 11. Roles, Permissions, and Access Control

### 11.1 Real access model

The current application does **not** implement conventional auth or RBAC.

There are no fully implemented concepts for:

* admin users
* staff users
* customer accounts
* organizations or tenant membership
* role tables
* auth provider sessions

### 11.2 What exists instead

#### Legal role

The user can be categorized as plaintiff or defendant. This affects workflow guidance, not system security.

#### Case owner

This is the real permission model. A browser-held token is hashed and associated with the case. The corresponding cookie is used to enforce case access.

#### AI beta user/tester

Some AI access is gated by beta-allowlist logic.

### 11.3 Practical implications

The current app supports **single-browser ownership of a case**, not true shared or collaborative access.

For a rebuild, this is a major architectural decision point:

* either preserve the lightweight owner-token model for a single-user tool, or
* replace it with real authentication, user accounts, and sharing/collaboration semantics

## 12. Core User Workflows

### Workflow 1: Start a case

1. User lands on `/start`
2. Selects county/court/role/category
3. App constructs case object
4. Case is saved server-side
5. Owner token/cookie is established

### Workflow 2: Complete structured intake

1. User opens `/intake-wizard`
2. Enters party, claim, and hearing information
3. Data is saved into structured `case_data`

### Workflow 3: Upload evidence

1. User opens `/documents`
2. Selects files and document type
3. Client chooses upload transport
4. Server stores file, metadata, text, and chunks when possible
5. OCR may be dispatched when needed

### Workflow 4: Review or tag evidence

1. User reviews document list
2. Opens document detail
3. Edits evidence metadata such as category or what the evidence supports
4. Optionally retries OCR

### Workflow 5: Ask AI about the case

1. User opens the global chat or chat page
2. Sends a question
3. Server loads case and document context
4. System returns deterministic or live AI answer grounded in case evidence

### Workflow 6: Check filing readiness

1. User opens filing guidance
2. App resolves forms and readiness state
3. User sees missing items or next actions

### Workflow 7: Manage drafts

1. User opens drafts
2. Creates or edits draft content
3. Drafts persist locally in IndexedDB

## 22. Migration Strategy Recommendations

### 22.1 What should be preserved

Preserve these elements unless there is a deliberate reason to replace them:

* the root Next.js app as the canonical product
* PostgreSQL-backed case/document/chunk model
* Vercel Blob-backed file storage pattern
* evidence-grounded chat design
* deterministic filing-readiness and forms-resolution logic
* chunk-based retrieval before adding vector complexity
* structured case JSON model, unless a more normalized schema is intentionally designed

### 22.2 What should be reworked or upgraded

Strongly consider replacing or improving:

* browser owner-token access model
* local-only draft storage
* local-only chat history
* incomplete OCR subsystem
* repo duplication and stale surfaces
* weak diagnostics/test coverage
* ad hoc rate limiting

### 22.3 Recommended target-state decisions

Before rebuilding, decide explicitly whether the new environment will remain:

#### Option A: single-user, case-owner model

Use a simplified owner-token approach again, but clean it up.

#### Option B: real authenticated product

Introduce:

* users
* sessions
* case ownership/sharing tables
* admin roles if needed
* collaborative access semantics

This is the single biggest product-architecture fork in the rebuild.

## 23. Recommended Rebuild Phasing

### Phase 1: establish clean canonical app shell

* isolate the root app as the basis for migration
* remove or quarantine legacy surfaces
* normalize layout/router structure
* confirm environment variables and deployment model

### Phase 2: rebuild core data and persistence

* re-create PostgreSQL schema from `ensureSchema`
* migrate case/document/chunk behavior
* preserve case JSON shape unless intentionally redesigning

### Phase 3: rebuild document ingestion

* implement direct upload
* implement blob-backed large-file upload
* preserve `.docx` and machine-readable PDF extraction behavior
* restore chunking pipeline

### Phase 4: complete OCR subsystem

* choose OCR provider or internal OCR path
* implement dispatch, retry, callback, monitoring, and failure handling cleanly
* ensure scanned PDFs become AI-readable where possible

### Phase 5: rebuild AI chat

* preserve server-side evidence grounding
* port deterministic response modes
* re-enable OpenAI integration
* improve retrieval only after baseline behavior is restored

### Phase 6: restore workflow pages

* start
* intake wizard
* dashboard
* documents
* filing guidance
* key dates
* drafts

### Phase 7: hardening

* tests
* observability
* better rate limiting
* auth/collaboration if required
* data migration utilities if old data must be preserved

## 26. Final Summary

The current THOXIE codebase has a coherent and valuable core. The root application already implements a meaningful server-backed workflow for California small claims, with structured case data, evidence uploads, document chunking, and grounded AI assistance. The rebuild should preserve that core rather than starting from the older prototypes.

The most important migration risks are:

* rebuilding from stale surfaces instead of the root app
* underestimating the importance of the document ingestion pipeline
* overlooking the incomplete OCR subsystem
* assuming the app already supports real multi-user roles when it does not

The most important strategic choice for the next environment is whether to keep the current lightweight owner-token model or replace it with real authentication and collaboration. That decision affects the schema, access model, UX, and deployment shape more than any other single factor.
